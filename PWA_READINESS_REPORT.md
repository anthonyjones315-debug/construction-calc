## 2. Service Worker Check

**Configuration:**

- **next.config.ts / next.config.js:** No PWA plugin. No `next-pwa`, `@serwist/next`, or Workbox configuration.
- **Registration:** `src/components/layout/ServiceWorker.tsx` registers `/sw.js` when `'serviceWorker' in navigator`.
- **SW source:** Served by **`src/app/sw.js/route.ts`** — a Route Handler that returns JavaScript with `Content-Type: application/javascript` and `Service-Worker-Allowed: /`.

**Behavior (from `sw.js/route.ts`):**

- **Cache name:** `bcp-v1`
- **Precache:** `["/", "/calculators", "/offline"]`
- **Install:** Opens cache, adds PRECACHE, then `skipWaiting()`
- **Activate:** Deletes other caches, then `clients.claim()`
- **Fetch:** Network-first for same-origin GET (and fonts); excludes `/api/`, `/auth/`. On network failure, serves from cache, then falls back to `/offline`.

No third-party PWA package is installed; the app uses a **custom inline Service Worker**.

## 4. Offline Fallback

**Offline page:** `src/app/offline/page.tsx` exists and is a static, self-contained “You’re offline” page with a link to `/calculators`.

**Routing:** `routes.offline` is `"/offline"`; the SW precache includes `"/offline"` and the fetch handler falls back to `caches.match("/offline")` when the request is not in cache.

**Result:** A proper offline fallback is in place for failed navigations.

## 6. Recommended Step-by-Step Implementation Plan

### Step 1: Fix manifest and icons (no new packages)

1. **Edit `public/site.webmanifest`:**
   - `"short_name": "Pro Calc"`
   - `"theme_color": "#FF8C00"`
   - `"background_color": "#0A0A0B"`
2. **Add real icon assets:**
   - Add `public/icon-192.png` and `public/icon-512.png` (maskable-safe art, e.g. logo with padding for safe zone).
   - In the manifest, set icon `src` to `"/icon-192.png"` and `"/icon-512.png"` (with correct `sizes` and `purpose`: `any` + `maskable`).
3. **Optional:** Add `app/icon.tsx` (or `app/icon.png`) to support Next.js metadata and `/icon`; then you can keep using `/icon?size=192` and `/icon?size=512` in the manifest if you prefer generated icons.

### Step 2: Choose a modern Next.js PWA package

**Recommendation: use `@serwist/next`** (successor to `next-pwa`, maintained and compatible with Next 14+ and App Router).

- **Why not `next-pwa`:** Largely unmaintained; can conflict with Next 14+ and Turbopack.
- **Why Serwist:**  
  - Workbox under the hood (precache, runtime caching, expiration).  
  - App Router–friendly (e.g. `serwist.config.ts` at project root).  
  - Supports custom `swSrc` or default generated SW, and **offline fallback** (e.g. `_offline` page).  
  - TypeScript, documented, and actively maintained.

**Alternative:** Continue with the current custom SW and only improve it (e.g. expand PRECACHE, add cache versioning and cleanup). That keeps the bundle free of a PWA dependency but leaves you responsible for correctness and updates.

### Step 3: Install and configure Serwist (when ready)

1. Install:  
   `npm i @serwist/next`  
   (and optionally `serwist` if you need the core build APIs.)
2. Add **`serwist.config.ts`** (or `.js`) at the project root:
   - Set `swDest` (e.g. `public/sw.js` or under `.next` and served via route).
   - Configure **precache**: include `/`, `/calculators`, `/offline`, and optionally key calculator routes (e.g. `/calculators/framing/wall-studs`).
   - Configure **runtime caching** for same-origin document/navigation requests (e.g. network-first with cache fallback) and optionally for static chunks.
   - Set **navigateFallback** to `/offline` (or your chosen offline URL) so unvisited routes show the offline page when offline.
3. In **next.config.ts**, wrap the config with `withSerwistInit` from `@serwist/next` and pass your `serwistConfig`.
4. **Registration:** Keep using your existing `ServiceWorker.tsx` but point it at the Serwist-generated SW path (e.g. `/sw.js` or the path you configured). Alternatively, use Serwist’s registration snippet if provided.

### Step 4: Align custom SW with Serwist (if migrating)

- Remove or disable the current **`src/app/sw.js/route.ts`** so only one SW is active (the one built by Serwist).
- Ensure the Serwist build outputs the SW to a URL that your app registers (e.g. `public/sw.js` and register `/sw.js`).
- After migration, run a production build and test install + offline behavior on a real device or Chrome DevTools.

### Step 5: Verify and re-audit

- **Lighthouse:** Run PWA audit; confirm manifest, icons, SW, and offline.
- **Chrome:** “Install” the app and test offline (e.g. disable network, reload, open `/calculators` and a few calculator pages).
- Re-check manifest in DevTools → Application → Manifest (name, short_name, theme_color, background_color, icons loading).
- Confirm **192×192** and **512×512** icons exist and that at least one is maskable.
